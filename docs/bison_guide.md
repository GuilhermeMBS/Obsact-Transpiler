# Fundamental Guide to Bison (GNU Parser Generator) 🐃

**Bison** is a general-purpose **Syntax Analyzer Generator** (also known as a *parser*). It converts a context-free grammar description into a C program that analyzes the structural layout of a token stream. While Flex groups characters into individual tokens (words), Bison verifies if the order of those tokens obeys the grammatical rules of the language (sentences).

---

## 🏗️ General Structure of a `.y` File

A Bison file is strictly structured into **three sections**, separated by the special delimiter `%%`. The layout mirrors the one found in Flex:

```yacc
Declarations and Token Definitions
%%
Grammar Rules and Semantic Actions
%%
Additional C Code
```

### 1. First Section: Declarations and Token Definitions
This section configures the C environment and declares the tokens that the parser expects to receive from the lexical analyzer (`yylex()`).

- **Pure C Block (`%{` and `%}`):** Anything written inside these delimiters is copied literally into the generated C parser file. This is where you put standard headers like `#include <stdio.h>` and declarations for global functions like `yyerror()` and `yylex()`.

- **Token Declarations (`%token`):** Defines the list of terminal symbols (tokens) that Flex will return. For example:

```
%token SET
%token NUMBER
```
- Operator Precedence (`%left`, `%right`, `%nonassoc`): Optional declarations used to resolve mathematical ambiguities (e.g., stating that multiplication has higher priority than addition).

### 2. Second Section: Grammar Rules (The Heart of Bison)
This section is where you implement the Context-Free Grammar (CFG) rules of your language. The syntax follows a Backus-Naur Form (BNF) variation:

```
non_terminal:
    terminal_one non_terminal_two    { c_action(); }
  | alternative_pattern              { alternative_c_action(); }
;
```
- **Left-Hand Side:** The non-terminal rule being defined (always lowercase by convention).

- **Right-Hand Side:** The sequence of tokens or other rules required to satisfy that rule.

- **Semantic Actions (`{ ... }`):** A block of C code executed immediately when the parser successfully matches that specific rule. This is where code generation, symbol table management, or interpretation occurs.

## 💡 Accessing Value Attributes (`$1`, `$2`, `$$`):
Bison allows you to pass data up the grammar tree using special variables:

- `$$` represents the return value of the current non-terminal rule.

- `$1` represents the value of the first symbol on the right-hand side, `$2` the second, and so on.

## 3. Third Section: Additional C Code
This section contains user-defined C functions. Bison requires at least one specific error-handling function implemented here:

```C
void yyerror(const char *s) {
    fprintf(stderr, "Syntax Error: %s\n", s);
}
```
You can also place the standard `main()` function here to trigger the parsing loop by calling `yyparse()`.

## 🧰 Key Functions and Variables
Bison provides and interacts with several critical hooks:

- `yyparse()` (function): The main entry point generated by Bison. When called, it repeatedly invokes `yylex()` to fetch tokens and parse the input stream until it reaches the End of File (EOF) or encounters an unrecoverable syntax error. It returns `0` on success and `1` on failure.

- `yyerror(const char* s)` (function): A user-defined error callback. Bison automatically calls this routine whenever it encounters an unexpected token that breaks the grammar rules.

- `yylval` (global variable): The communication bridge for token values. When Flex matches a token (like a specific number), it stores the actual value inside `yylval` right before returning the token ID to Bison.

## 📟 Main Compilation Commands
To build a parser using Bison, you process the `.y` description file into C code and link it against your scanner:

**1. Generate the C source and header files:**

```
win_bison -d sintatico.y
```
*The `-d` flag is crucial. It tells Bison to generate both the source file (`syntax.tab.c`) and a header file (`syntax.tab.h`). This header file contains the token definitions and must be included in your Flex file (`lexical.l`) so that both components agree on token numeric IDs.*

**2. Compile the engine components with GCC:**

```
gcc sintatico.tab.c lex.yy.c -o compiler
```
*This compiles the parser alongside the lexer to create the final executable transpiler.*
