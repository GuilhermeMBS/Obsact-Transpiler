# Fundamental Guide to Flex (Fast Lexical Analyzer Generator) 📜

**Flex** is a tool used to generate **Lexical Analyzers** (also known as *scanners* or *tokenizers*). Its primary role is to read an input stream of characters (source code), identify patterns using **Regular Expressions (Regex)**, and convert those patterns into meaningful units called **Tokens**.

---

## 🏗️ General Structure of a `.l` File

A Flex file is strictly divided into **three sections**, separated by the special delimiter `%%`. The basic skeleton follows this layout:

```lex
Definitions and Declarations
%%
Matching Rules (Regex & Actions)
%%
Additional C Code (Auxiliary Functions)
```

### 1. First Section: Definitions and Declarations
This section is used to configure the environment and prepare the generated C code. It accepts two main blocks:

- **Pure C Block (`%{` and `%}`):** Everything placed inside these delimiters is copied literally to the top of the C file generated by Flex. This is where you place your `#include <stdio.h>`, `#include "syntax.tab.h"` (to communicate with Bison), and global variables.

- **Regular Definitions:** Shortcuts for complex regular expressions. If you don't want to repeat `[0-9]+` throughout your code, you can create an alias here:

```
DIGIT    [0-9]
NUMBER   {DIGIT}+
```

### 2. Second Section: Rules (The Heart of Flex)
This section defines which text patterns your analyzer should look for and what it should do when it encounters each one. The standard syntax is:

```
REGEX_PATTERN      { c_action(); }
```

- **Pattern (Left Side):** A classic regular expression that attempts to match the input text.

- **Action (Right Side):** A block of C code that executes the exact moment the pattern is recognized. Usually, this action performs a `return TOKEN;` to inform the syntax analyzer (Bison) which token was found.

⚠️ **Flex's Golden Rule:** If the input text matches more than one rule at the same time, Flex resolves the ambiguity using two priority criteria:

1. It chooses the rule that matches the **longest number of characters** (longest match).

2. If the number of characters is equal, it chooses the rule that was **written first** in the file.

### 3. Third Section: Additional C Code
This section is optional. It is used to implement the main() function (if you are testing Flex in isolation, without Bison) or utility functions you want to create to manipulate data within your compiler. Everything written here is copied entirely to the end of the generated C file.

## 🧰 Special Global Variables and Functions
Flex automatically provides a set of internal variables and functions in C that you can use inside your actions:

- `yytext` (type `char*`): A string containing the exact text that was just matched by the regular expression. If the regex was `[a-z]+` and the user typed `temperature`, `yytext` will hold the string `"temperature"`.

- `yyleng` (type `int`): Stores the length (number of characters) of the text matched in `yytext`.

- `yylex()` (function): The main function generated by Flex. When Bison (or your `main`) calls `yylex()`, Flex starts reading the input and only stops when it finds a valid token and executes a `return` action.

- `yyin` (type `FILE*`): A standard C file pointer. By default, Flex reads from the standard input (`stdin` - the keyboard). If you want it to read from an `.obsact` file, simply open the file in C and assign it to this variable: `yyin = fopen("test.obsact", "r");`.

- `yywrap()` (function): A utility function called by Flex when it reaches the End of File (EOF). If it returns `1`, Flex terminates the analysis. If it returns `0`, Flex assumes you opened another file in `yyin` and continues scanning.

## 📟 Main Compilation Commands
To transform your descriptive `.l` file into an executable binary, the standard terminal workflow involves Flex and the GCC compiler:

**1. Generate the C file from Flex:**

```
win_flex lexico.l
```
*This reads your `lexico.l` and outputs a pure C source file named `lex.yy.c`.*

**2. Compile the generated file along with the rest of the project:**

```
gcc lex.yy.c -o analyzer
```
*GCC compiles the engine generated by Flex and creates the final executable.*

## 💡 Quick Regex Syntax Reference for Flex
- `"word"`: Matches the exact string inside the quotes (excellent for reserved keywords like `se`, `entao`, `dispositivo`).

- `[a-z]`: Matches any lowercase letter.

- `[a-zA-Z]`: Matches any uppercase or lowercase letter.

- `+`: Signifies "one or more occurrences" of the preceding element.

- `*`: Signifies "zero or more occurrences" of the preceding element.

- `\n`, `\t`, ` `: Represents newlines, tabs, and whitespaces. We usually use an empty action `{ /* ignore */ }` to discard these characters.

- `.`: The dot in Regex matches **any character** except a newline. It is commonly used as the very last rule of the file to catch lexical errors (invalid characters that didn't match any previous rule).
